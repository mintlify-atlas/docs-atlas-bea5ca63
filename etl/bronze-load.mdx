## Overview

The `bronze.load_bronze()` procedure is responsible for ingesting raw data from CSV files into the Bronze layer of the data warehouse. This layer stores data in its original format without any transformations, providing a historical archive of source data.

<Note>
  The Bronze layer follows the "load first, ask questions later" philosophy. All data is loaded as-is, preserving the exact source format for auditability and reprocessing capabilities.
</Note>

## Procedure Code

Here's the complete `bronze.load_bronze()` procedure from `/home/daytona/workspace/source/scripts/bronze/proc_load_bronze.sql`:

```sql
CREATE OR REPLACE PROCEDURE bronze.load_bronze()
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_step TEXT;
  v_ok int := 0;
  v_err int := 0;
  v_msg TEXT;
  v_start_batch timestamp := clock_timestamp();
  v_end_batch timestamp;
BEGIN
  RAISE NOTICE '[%] load bronze started', to_char(clock_timestamp(), 'HH24:MI:SS');
  
  -- Load CRM customer information
  BEGIN
    DECLARE
      v_start_date timestamp  := clock_timestamp();
    BEGIN
    v_step := 'load bronze.crm_cust_info';
    RAISE NOTICE 'Step [%] started at %', v_step, to_char(v_start_date, 'HH24:MI:SS');
    TRUNCATE TABLE bronze.crm_cust_info;
    COPY bronze.crm_cust_info
      FROM '/datasets/source_crm/cust_info.csv'
      WITH (FORMAT csv, HEADER true);
    v_ok := v_ok + 1;
    RAISE NOTICE 'Step [%] completed successfully (%.0f s)',
             v_step,
             extract(epoch FROM (clock_timestamp() - v_start_date));
    Exception WHEN OTHERS THEN
      v_err := v_err + 1;
      v_msg := 'Error in step: ' || v_step || ' - ' || SQLERRM;
      RAISE EXCEPTION '%', v_msg;
  END;
  END;

  -- Load CRM product information
  BEGIN
  DECLARE
      v_start_date timestamp  := clock_timestamp();
    BEGIN
    v_step := 'load bronze.crm_prd_info';
    TRUNCATE TABLE bronze.crm_prd_info;

    v_step := 'load bronze.crm_sales_details';
    TRUNCATE TABLE bronze.crm_sales_details;
```

## Data Loading Process

<Steps>
  <Step title="Truncate Existing Data">
    Each table is truncated before loading to ensure a clean slate:
    ```sql
    TRUNCATE TABLE bronze.crm_cust_info;
    ```
  </Step>
  
  <Step title="Execute COPY Command">
    PostgreSQL COPY command loads CSV data directly from the file system:
    ```sql
    COPY bronze.crm_cust_info
      FROM '/datasets/source_crm/cust_info.csv'
      WITH (FORMAT csv, HEADER true);
    ```
  </Step>
  
  <Step title="Track Success">
    Successful loads increment the counter and log execution time:
    ```sql
    v_ok := v_ok + 1;
    RAISE NOTICE 'Step [%] completed successfully (%.0f s)', v_step, duration;
    ```
  </Step>
</Steps>

## COPY Command Details

The procedure uses PostgreSQL's COPY command with these parameters:

<Accordion title="FORMAT csv">
  Specifies CSV (Comma-Separated Values) format for input files.
</Accordion>

<Accordion title="HEADER true">
  Indicates that the first line of each CSV file contains column headers and should be skipped during data load.
</Accordion>

### COPY Command Syntax

```sql
COPY table_name
  FROM 'file_path'
  WITH (FORMAT csv, HEADER true);
```

<Note>
  The COPY command requires superuser privileges or appropriate file system permissions. The procedure uses `SECURITY DEFINER` to run with elevated privileges.
</Note>

## Error Handling

### Exception Pattern

Each load step is wrapped in an exception handler:

```sql
EXCEPTION WHEN OTHERS THEN
  v_err := v_err + 1;
  v_msg := 'Error in step: ' || v_step || ' - ' || SQLERRM;
  RAISE EXCEPTION '%', v_msg;
```

### Common Errors

<Accordion title="File Not Found">
  **Error**: `could not open file for reading: No such file or directory`
  
  **Solution**: Verify that CSV files exist in the specified paths:
  - `/datasets/source_crm/`
  - `/datasets/source_erp/`
</Accordion>

<Accordion title="Permission Denied">
  **Error**: `could not open file for reading: Permission denied`
  
  **Solution**: Ensure PostgreSQL process has read permissions on the CSV files and directory.
</Accordion>

<Accordion title="Invalid CSV Format">
  **Error**: `invalid input syntax for type`, `extra data after last expected column`
  
  **Solution**: Verify CSV file format matches table schema. Check for:
  - Correct number of columns
  - Proper delimiter (comma)
  - Valid data types
  - Proper escaping of special characters
</Accordion>
